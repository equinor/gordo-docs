CLI¶

gordo-components CLI¶

Available CLIs for Gordo:

gordo-components¶

The main entry point for the CLI interface

gordo-components [OPTIONS] COMMAND [ARGS]...

build¶

Build a model and deposit it into ‘output_dir’ given the appropriate config settings.

Parameters
———-
output_dir: str
Directory to save model & metadata to.
model_config: str
String containing a yaml which will be parsed to a dict which will be used in
initializing the model. Should also contain key ‘type’ which references the
model to use. ie. KerasAutoEncoder
data_config: dict
kwargs to be used in intializing the dataset. Should also
contain kwarg ‘type’ which references the dataset to use. ie. InfluxBackedDataset
metadata: dict
Any additional metadata to save under the key ‘user-defined’
model_register_dir: path
Path to a directory which will index existing models and their locations, used
for re-using old models instead of rebuilding them. If omitted then always
rebuild
print_cv_scores: bool
Print cross validation scores to stdout
model_parameter: List[Tuple]
List of model key-values, wheres the values will be injected into the model
config wherever there is a jinja variable with the key.
gordo-components build [OPTIONS] [OUTPUT_DIR] [MODEL_CONFIG] [DATA_CONFIG]

Options

--metadata <metadata>¶
--model-register-dir <model_register_dir>¶
--print-cv-scores¶

Prints CV scores to stdout

--model-parameter <model_parameter>¶

Key-Value pair for a model parameter and its value, may use this option multiple times. Separate key,valye by a comma. ie: –model-parameter key,val –model-parameter some_key,some_value

client¶

Entry sub-command for client related activities

gordo-components client [OPTIONS] COMMAND [ARGS]...

Options

--project <project>¶

The project to target

--target <target>¶

Single target, instead of all project targets

--host <host>¶

The host the server is running on

--port <port>¶

Port the server is running on

--scheme <scheme>¶

tcp/http/https

--gordo-version <gordo_version>¶

Version of gordo

--batch-size <batch_size>¶

How many samples to send

--parallelism <parallelism>¶

Maximum asynchronous jobs to run

--metadata <metadata>¶

Key-Value pair to be entered as metadata labels, may use this option multiple times. to be separated by a comma. ie: –metadata key,val –metadata some key,some value

predict¶

Run some predictions against the target

gordo-components client predict [OPTIONS] START END

Options

--data-provider <data_provider>¶

DataProvider dict

--output-dir <output_dir>¶

Save output prediction dataframes in a directory

--influx-uri <influx_uri>¶

Format: <username>:<password>@<host>:<port>/<optional-path>/<db_name>

--influx-api-key <influx_api_key>¶

Key to provide to the destination influx

--influx-recreate-db¶

Recreate the desintation DB before writing

--forward-resampled-sensors¶

forward the resampled sensor values

--ignore-unhealthy-targets¶

Ignore any unhealthy targets. By default the client will raise an error if any unhealthy endpoints are encountered.

run-watchman¶

Start the Gordo Watchman server for this project. Which is responsible for dynamically comparing expected URLs derived from a project config fle against those actually deployed to determine and report their health.

Must have the following environment variables set:
PROJECT_NAME: project_name for the config file
TARGET_NAMES: A list of non-sanitized machine / target names
gordo-components run-watchman [OPTIONS] PROJECT_NAME PROJECT_VERSION
                              TARGET_NAMES

Options

--host <host>¶

The host to run the server on.

--port <port>¶

The port to run the server on.

--debug <debug>¶

Run in debug mode

--namespace <namespace>¶

Namespace watchman should make requests in for ML servers

--ambassador-namespace <ambassador_namespace>¶

Namespace watchman expects Ambassador to be in
